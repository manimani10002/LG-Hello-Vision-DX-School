## 💡 [37일차] API 및 Open API의 이해

### 1. Library vs API: 어떻게 다른가?

가장 헷갈리기 쉬운 개념을 잘 구분하셨습니다.

* **Library (도서관):** 요리법(코드) 자체를 내 컴퓨터로 빌려오는 것과 같습니다. 내가 직접 그 코드를 실행하고 제어합니다.
* **API (키오스크/점원):** 요리법은 몰라도 됩니다. 점원에게 "치즈버거 하나 주세요(Request)"라고 말하면, 주방(서버)에서 만들어진 결과물(Response)을 가져다줍니다. 나는 **'기능'**만 이용할 뿐 내부 로직은 알 필요가 없습니다.
* **Open API:** 누구나 사용할 수 있도록 공공기관이나 기업(구글, 카카오 등)에서 공개한 API입니다.



---

### 2. HTTP를 통한 정보 교환 메커니즘

API 통신은 기본적으로 **요청(Request)**과 **응답(Response)**의 쌍으로 이루어집니다.

* **Request (요청):** 클라이언트(나)가 서버에 "이 데이터를 줘"라고 보내는 메시지.
* **Response (응답):** 서버가 요청을 처리한 후 주는 결과물. (주로 JSON이나 XML 형식)

#### 📝 요청의 구성 요소

1. **URL (Endpoint):** 목적지 주소입니다. (예: `https://api.weather.com/today`)
2. **Method:** 어떤 행위를 할 것인지 정의합니다. (GET, POST 등)
3. **Parameter (매개변수):** 요청에 필요한 구체적인 정보입니다. (예: `city=seoul`, `date=20260206`)

---

### 3. 데이터 전송 방식: GET vs POST

데이터를 보낼 때 어디에 담느냐에 따라 크게 두 가지로 나뉩니다.

| 구분 | GET | POST |
| --- | --- | --- |
| **데이터 위치** | **URL** 뒤에 붙여서 보냄 (Query String) | HTTP 메시지의 **Body**에 숨겨서 보냄 |
| **보안성** | 주소창에 다 보임 (낮음) | 주소창에 보이지 않음 (상대적 높음) |
| **데이터 용량** | 길이 제한이 있음 (짧은 데이터) | 용량 제한이 거의 없음 (대용량/파일) |
| **주 목적** | 데이터 **조회(Read)** | 데이터 **생성/변경(Write)** |
| **예시** | 검색어 입력, 페이지 번호 클릭 | 로그인(ID/PW), 게시글 작성 |

---

### 4. URL과 Body에 담는 방식 (상세 설명)

#### ① URL에 담는 방식 (주로 GET)

* 형식: `주소?키1=값1&키2=값2`
* 예: `https://search.naver.com/search.naver?query=날씨`
* **특징:** 링크를 복사해서 남에게 줬을 때 동일한 결과 화면을 볼 수 있습니다(캐싱 가능).

#### ② Body에 담는 방식 (주로 POST)

* 눈에 보이지 않는 패킷의 '몸통(Body)' 안에 데이터를 실어 보냅니다.
* **특징:** 로그인 정보처럼 노출되면 안 되는 민감한 정보나, 이미지 파일 같은 큰 데이터를 보낼 때 사용합니다.

---

### 5. API 파라미터(Parameter)의 종류

* **Path Variable:** URL 경로의 일부로 사용 (예: `/users/123` → 123번 유저)
* **Query Parameter:** URL 끝에 `?`로 연결 (예: `/search?q=apple`)
* **Request Body:** JSON 형식 등으로 Body에 담김 (예: `{"id": "test", "pw": "1234"}`)

---

**오늘의 학습 요약:**
API는 복잡한 내부 로직을 몰라도 **규격(HTTP)**에 맞춰 **매개변수(Parameter)**를 던지면 원하는 결과를 얻을 수 있는 효율적인 도구입니다. 데이터의 성격에 따라 **GET**과 **POST**를 적절히 선택하는 것이 실무의 핵심입니다.
